# ADR-0005: Coverage discipline

## Status
Accepted (documented retroactively, v4.0.2)

## Context
Before the ports were introduced, much of the subprocess- and DB-bound logic was
untestable, leaving line coverage around 54%. The build also carried no-op
`#[cfg(not(tarpaulin_include))]` markers that were silently ineffective under the
actual coverage tool in use (`cargo-llvm-cov`), giving a false sense of
exclusion. Coverage needed to become both higher and honestly measured.

## Decision
- Use the test seams from ADR-0002 (`CommandRunner` + `FakeRunner`) and ADR-0003
  (`MultiConnection` + in-memory SQLite) to make previously untestable paths
  unit-testable. This raised line coverage **54% → 84%**.
- Replace the ineffective `#[cfg(not(tarpaulin_include))]` markers with
  `#[coverage(off)]` gated on the `coverage_nightly` cfg (set by `cargo-llvm-cov`;
  requires the nightly `coverage_attribute` feature, enabled in `main.rs`/`lib.rs`).
- Exclude generated `schema.rs` from the CI `llvm-cov` report.
- Run tests under `cargo-nextest` (process-per-test), which makes the global
  `GLOBAL_VAR` `OnceCell` deterministic (each test gets a fresh process).

## Alternatives considered
- **Keep tarpaulin markers** — rejected: silently ineffective under `llvm-cov`;
  they exclude nothing and mislead readers into thinking coverage is curated.
- **Refactor away `GLOBAL_VAR` to fix threaded-test flakiness instead of using
  nextest** — deferred: larger change (see brief Open Questions); nextest's
  process isolation resolves the test-determinism symptom now at near-zero cost.

## Consequences
- Positive: honest, higher coverage (84%); exclusions actually take effect under
  the real tool; deterministic test runs.
- Negative: the toolchain is pinned to **nightly** (for `coverage_attribute`);
  test determinism depends on running via nextest — a plain threaded
  `cargo test` can still flake because of the process-global `OnceCell`.
