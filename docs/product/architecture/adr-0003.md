# ADR-0003: diesel MultiConnection for backend-agnostic persistence

## Status
Accepted (documented retroactively, v4.0.2)

## Context
Production stores CV metadata in Postgres (reached over Tailscale). Tests must
run without a live Postgres server and must be deterministic in CI. The query
code (insert / idempotent lookup / list) should be written once and run against
both backends.

## Decision
Use diesel's `#[derive(diesel::MultiConnection)]` to define a backend-agnostic
connection enum:

```rust
#[derive(diesel::MultiConnection)]
pub enum DbConnection {
    Postgresql(PgConnection),
    Sqlite(SqliteConnection),
}
```

All persistence functions take `&mut DbConnection`. `establish_connection`
picks the variant from the configured `[db] engine` (Postgres in prod, SQLite —
including in-memory `:memory:` — in tests). Because `MultiConnection` does **not**
support `Selectable::as_select` / `as_returning`, queries rely on the default
all-columns selection; the `Cv` struct's field order is kept aligned with the
`cv` table so default selection deserializes correctly.

## Alternatives considered
- **Postgres-only** — rejected: untestable without a running server; couples CI
  to network + Tailscale + credentials.
- **Generic-over-`Connection` trait bounds** — rejected: every query function
  would need verbose `where C: Connection<...>` bounds and backend-specific
  trait constraints; `MultiConnection` encapsulates this in one enum.

## Consequences
- Positive: one query codebase for both backends; fast, deterministic tests on
  in-memory SQLite; trivial local runs without Postgres.
- Negative: loss of `as_select`/`as_returning` ergonomics and their
  compile-time column-mapping safety — correctness now depends on keeping `Cv`
  field order in sync with the schema (documented in `database.rs`).
