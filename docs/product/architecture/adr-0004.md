# ADR-0004: Pre-usage tool-availability checks

## Status
Accepted (documented retroactively, v4.0.2)

## Context
The tool depends on external binaries that may be absent from `PATH` when the
program is not run inside the devenv shell: `just` and `tectonic` (build),
`pdf_viewer`/zathura (view), `sudo` + `tailscale` (Postgres reachability probe).
If these are missing, a raw subprocess failure surfaces as a cryptic error far
from the actual cause, with no guidance on how to fix it.

## Decision
Check tool availability **before use**, at the orchestration layer.
`ensure_tools_available(tools)` scans `PATH` via `tool_on_path` and, on any
missing tool, returns an error listing the missing binaries plus a `DEVENV_HINT`
telling the user to run inside `devenv shell`.

Probe sites:
- `prepare_cv` checks `[builder, "tectonic"]` before building (the builder drives
  tectonic through the Justfile);
- `main` checks the configured `pdf_viewer` before viewing;
- `check_if_db_env_is_set_or_set_from_config` checks `["sudo", "tailscale"]`
  before the Postgres/Tailscale path, then probes `tailscale status`.

This is the **Earned Trust** principle applied to environment dependencies: the
program demonstrates the substrate can honor the contract before relying on it,
and refuses to proceed with a structured, actionable error otherwise.

## Alternatives considered
- **Let the subprocess fail with a cryptic error** — rejected: poor diagnosability;
  the failure is detached from its cause and gives the user no remediation path.
- **Check once at startup for all tools** — rejected as unnecessary: not every
  action needs every tool (e.g. list/remove need neither `just` nor `tectonic`);
  per-action checks keep the probe scoped to what will actually run.

## Consequences
- Positive: fast, fail-fast errors naming the missing tool and the devenv remedy;
  checks live where the dependency is about to be used.
- Negative: a small amount of duplicated check wiring across call sites; PATH
  checks are advisory (a tool present but broken still fails at run — acceptable,
  since the subprocess error then carries real diagnostics).
