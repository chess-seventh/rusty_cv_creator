# ADR-0001: Variant-based CV build via Justfile

## Status
Accepted (documented retroactively, v4.0.2)

## Context
The previous flow copied a single LaTeX template, compiled it with `xelatex`,
and substituted a `BLANKPOSITION` placeholder to inject the job title. This gave
one generic CV shape per application and coupled the binary to a specific LaTeX
engine and a brittle string-replacement step.

Job applications need genuinely different CV framings (DevOps, Platform, SRE,
Engineering Manager), not just a swapped title. The template repository
(`git@github.com:chess-seventh/cv.git`) already expresses these as **4 variants**
built by a `Justfile` recipe using `tectonic`.

## Decision
Build a selected one of 4 variants (`senior-devops`,
`senior-platform-engineer`, `senior-sre`, `engineering-manager`) by invoking a
config-driven `just build <variant>` inside a dated working copy of the template.

- **Variant resolution precedence** (`resolve_variant`):
  1. explicit `--variant` flag (validated against `CV_VARIANTS`);
  2. keyword inference from the job title (`infer_variant_from_job_title`) —
     **`engineering-manager` is matched first** because manager titles also
     contain "DevOps"/"Platform";
  3. configured `[variant] default` (fallback `senior-devops`).
- Output prefix, builder, and recipe come from the INI (`BuildConfig`): no
  hard-coded engine or output directory. `just build <variant>` emits
  `<prefix>-<variant>.pdf` next to the driver file.

## Alternatives considered
- **Keep `xelatex` + `BLANKPOSITION` placeholder** — rejected: brittle string
  substitution, single CV shape, engine lock-in, no per-role tailoring.
- **Inference-only (no flag)** — rejected: user cannot override a misclassified
  title (e.g. ambiguous or novel titles).
- **Flag-only (no inference)** — rejected: forces the user to always specify a
  variant even when the title makes it obvious; poor ergonomics.

## Consequences
- Positive: per-role tailoring; build engine/recipe owned by the template repo
  and INI, not the binary; explicit override plus sensible inference; pure,
  unit-tested resolution logic.
- Negative: the binary now depends on the template repo's recipe-name and
  output-naming contract (`just build <variant>` → `<prefix>-<variant>.pdf`);
  drift there breaks builds. Mitigated by the per-variant smoke test recommended
  in the brief and the pre-usage tool checks (ADR-0004).
