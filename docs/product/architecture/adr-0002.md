# ADR-0002: CommandRunner port for external processes

## Status
Accepted (documented retroactively, v4.0.2)

## Context
The build, PDF-view, and Tailscale-status flows all shell out to external
programs. Calling `std::process::Command` directly inside business logic makes
those code paths effectively untestable: tests would have to really run `just`,
`tectonic`, `zathura`, or `sudo tailscale`, which is non-deterministic and
unavailable in CI. Testability is the primary quality driver for this codebase.

## Decision
Introduce a driven port `CommandRunner` (trait) with three methods:
- `status(program, args, cwd) -> io::Result<bool>` — run to completion;
- `output(program, args) -> io::Result<(bool, String)>` — capture stdout;
- `spawn(program, args) -> io::Result<()>` — detached launch (PDF viewer).

Two adapters: `SystemRunner` (production, backed by `std::process::Command`) and
`testing::FakeRunner` (records invocations, returns canned success/stdout/io
errors). The runner is injected as `&dyn CommandRunner` into `compile_cv`,
`view_cv_file`, and `is_tailscale_connected`.

## Alternatives considered
- **Call `std::process::Command` directly** — rejected: untestable without real
  external tools; non-deterministic; cannot assert the exact command invoked.
- **Generic `<R: CommandRunner>` bounds instead of `&dyn`** — viable but adds
  monomorphization/signature noise for no measurable benefit at this scale;
  dynamic dispatch is negligible for subprocess-bound work.

## Consequences
- Positive: subprocess logic is unit-tested with fakes; tests assert exact
  command strings (e.g. `"just build senior-devops"`); production behavior is
  isolated in one small adapter.
- Negative: one extra indirection layer; the port must be threaded through
  call sites that perform subprocess effects.
